# 真实终端回归清单

## 目的

这份文档记录 `codex-auto` 的交互式终端回归要求。

它不是单元测试，也不是普通的无 TTY 集成测试；行业里更接近：

- 交互式端到端回归
- 手工验收回归
- 终端 UI / TTY 回归测试

当代码涉及真实终端显示、输入、PTY 托管、交互切号时，必须执行这里的检查。

## 何时必须执行

出现以下任一情况时，必须做真实终端回归：

- 修改 `src/lib/session.ts`
- 修改 PTY、TTY、shell 启动、raw mode、resize、控制序列处理
- 修改交互模式下的 quota 检测、自动切号、恢复逻辑
- 修复分屏、中文输入、重绘错乱、提示符异常、状态行残影等问题

## 执行前准备

1. 先运行自动化测试：

```bash
npm test
```

2. 构建最新产物：

```bash
npm run build
```

3. 确保真实终端里执行的是最新版本，而不是旧全局安装：

```bash
npm install -g .
```

## 必测场景

### 1. 启动无控制序列泄漏

步骤：

```bash
codex-auto
```

验收：

- 启动后不应出现字面量 `^[[...`、`^[]...`、`[` 之类控制序列
- 不应出现终端查询响应被直接打印到屏幕
- 首屏应正常显示 Codex UI

### 2. 普通输入回显正常

步骤：

- 在提示符输入短文本，例如 `123`
- 回车发送

验收：

- 输入内容应正常显示在输入区
- 不应出现乱序回显、重复回显、半截控制字符

### 3. 中文输入

步骤：

- 输入中文，例如 `你好`
- 再测试中英文混排，例如 `你好 hi 123`

验收：

- 不应出现错位、残影、光标漂移
- 中文宽字符不应把布局撑坏

### 4. 分屏场景

步骤：

- 在真实终端里分屏或缩小 pane 宽度
- 再次启动 `codex-auto`
- 输入短文本并观察动态界面刷新

验收：

- 不应出现脏屏、重复绘制、提示符错位
- 动态刷新区域不应把旧内容残留在屏幕上

### 5. 动态状态行刷新

步骤：

- 观察启动阶段、MCP 启动阶段、`Working...` 一类动态状态变化

验收：

- 状态行更新应平滑
- 不应残留旧帧内容
- 不应把刷新控制逻辑泄漏成字面量文本

### 6. 交互式自动切号与恢复

适用条件：

- 只要本次改动触及交互模式下的额度检测、切号、恢复，就必须测

建议方式：

- 使用仓库现有可控 fixture 或可复现账号环境，让第一账号触发 quota
- 观察是否自动切到下一个账号并恢复会话

验收：

- 能检测到明确 quota 输出
- 能自动切到下一个账号
- 能继续 `resume` 到同一会话
- 切号后终端界面仍然不乱

### 7. 同项目双终端并发切号不串会话

适用条件：

- 只要本次改动触及交互模式下的会话绑定、切号、恢复，就必须测

步骤：

- 在同一个项目目录里打开两个真实终端
- 分别启动两个独立的 `codex-auto` 交互会话
- 让两个会话都各自产生可区分的上下文
- 只让其中一个会话触发 quota 并观察自动切号恢复

验收：

- 触发 quota 的那个终端只能恢复到它自己原来的会话
- 另一个终端的会话不能被抢占或串接
- 切号恢复后界面不乱、输入正常

### 8. 跨项目双终端并发切号不串会话

适用条件：

- 只要本次改动触及交互模式下的会话绑定、切号、恢复，就必须测

步骤：

- 在两个不同项目目录里分别打开真实终端
- 各自启动独立的 `codex-auto` 交互会话
- 让其中一个会话触发 quota 并观察自动切号恢复

验收：

- 触发 quota 的终端只能恢复到它自己的原会话
- 另一个项目中的会话不能被误恢复或误影响
- 切号恢复后界面不乱、输入正常
